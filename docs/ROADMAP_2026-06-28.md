## Current Milestone Verdict

The prototype has crossed from a basic match-3 loop into a readable mobile RPG duel slice. The player can now see what their move does, what Shade wants, when a risky shade move hurts Aurora, and how spells target the board.

The next problem is no longer raw feature absence. It is whether the duel repeatedly creates understandable, fair, tunable decisions.

The first 500-seed simulation showed that spells are not weak when used rationally: Aurora win rate moved from 76.8% with greedy swaps to 98.4% with spell-aware action selection. Human logs still show spell confusion, so the next user-facing work should make spell outcomes predictable before tuning costs or effects.

## Next Milestones

1. **Aurora/Shade Character Fantasy Pass**
   Continue from `docs/COMBAT_READ_COPY_PASS_2026-06-28.md`: strengthen the two sides as fantasy forces, not just colored UI panels. First candidates: side titles, faction marks, stronger critical/braced/charging states, and more distinctive victory/defeat result drama.

2. **Simulation v2 + Balance Review**
   Extend `scripts/sim-battles.ts` with spell breakdowns, damage split, extra-turn chain histogram, loss seed samples, and failure signatures. Record findings in `docs/BALANCE_REVIEW_2026-06-28.md` or a follow-up.

3. **Mobile WebApp Shell Polish**
   Keep testing the glued shell on shorter mobile heights. The current shell is full-bleed and more cockpit-like; remaining work is mainly safe-area validation in Telegram WebApp/browser chrome.

4. **Spell Preview Refinement**
   Keep the new target preview compact on mobile, then refine the labels after playtesting. First candidates: distinguish net mana from gross mana, make Crown Strike row collection easier to parse, and test whether action labels such as Create sun / +4 Guard / Clear row are enough without longer target text.

5. **Battle Recap Polish**
   Playtest whether the recap's cause, turning point, pressure metrics, and one-line lesson are understandable on a real phone. If needed, shorten copy before adding more post-fight detail.

6. **Enemy Profile / Affinity Layer**
   Move Shade Knight toward an explicit enemy profile rather than a generic scoring model. First target: visible preferences such as shade damage, extra turns, guard denial, and lethal setup.

7. **Physical Phone QA**
   Test the public build on a real 390px-class phone browser after rules and spell preview stabilize. Validate drag direction, tap-tap fallback, viewport height, and whether combat text remains readable with browser chrome.

## Defer

- Campaign map, relic inventory, multiple enemies, and progression.
- Wildcards or random mana surge mechanics.
- Heavy VFX and animation passes before battle balance is understood.
- New asset generation unless physical QA shows a specific readability failure.
- More spells before current spell value is measurable and understandable.
- Enemy/content expansion before the single-duel loop is tunable.

## Immediate Recommendation

Run a real-phone shell review before more balance or feature work. The board is still the strongest object; the surrounding UI is now more console-like, but real device screenshots should decide whether the shell feels premium or merely tighter.
